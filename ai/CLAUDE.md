# AI 개발 및 운영 통합 지침 (Unified Dev Protocol)

## 1. 기본 원칙 (Global Rules)
CONVENTIONS.md 를 확인한 후 준비가 완료되면 보고해. 모든 최종 산출물(대화, 코드 주석, 문서 등)은 자연스러운 한국어로 작성한다. 코드 식별자(변수, 함수명 등)는 영어를 사용하되, 이에 대한 설명과 주석은 예외 없이 한국어여야 한다.

**사전 참조**
작업 착수 전 반드시 `.ai/CONVENTIONS.md`와 `.ai/SPEC.md``.ai/STORY.md`를 확인하고 그 내용을 준수한다.
- 파일이 존재하지 않을 경우: 사용자에게 해당 사실을 즉시 알리고, 새로 생성할지 작업을 대기할지 확인을 받은 후 진행한다.

**지시 우선순위**
충돌 발생 시 아래 계층 순서를 따른다.
```
사용자의 명시적 지시 > SPEC.md > CONVENTIONS.md > 본 문서의 기본값
```

**태도**
건조하고 직관적이며 간결한 어조를 유지한다. 아첨이나 불필요한 수사를 배제하고 객관적 사실과 논리에 근거하여 소통한다.

## 2. 표준 작업 절차 (SOP)

모든 요청은 다음 5단계를 엄격히 준수하여 수행한다.

**[1. Context Analysis]**
현재 코드 상태와 문서의 맥락을 완벽히 파악한다. 사전 참조 파일의 존재 여부를 이 단계에서 확인한다.

**[2. Plan]**
파일 생성/수정/삭제 계획을 구체적인 리스트로 작성한다. 변경의 이유와 구현 방법을 명확히 정의한다.

**[3. Execute]**
수립된 계획을 100% 이행한다. 문법 오류를 방지하고 파일 경로 및 스코프를 엄격히 준수한다.

**[4. Review]**
작성된 결과물을 비판적으로 검토한다. 보안, 성능, 로직 결함, 컨벤션 준수 여부를 확인하고 스스로 개선안을 도출한다.
- 출력 범위: 사용자 확인이 필요한 항목(잠재적 결함, 설계 선택의 트레이드오프 등)만 간결히 보고한다. 내부적으로 해결된 사항은 보고하지 않는다.

**[5. Verify]** *(해당 시 적용)*
단위 테스트 작성 또는 실행 가능 여부를 확인한다. 테스트 환경이 없는 경우 경계값(edge case)과 예외 처리 흐름을 코드 레벨에서 명시적으로 서술한다.

## 3. 코딩 컨벤션 (Coding Conventions)

### 식별자 명명 규칙

| 대상 | 규칙 | 예시 |
|------|------|------|
| Class | `PascalCase` | `WorldSystem` |
| Variable / Function | `camelCase` | `isDragging`, `createFloor` |
| Constant | `UPPER_SNAKE_CASE` | `PANEL_COUNT` |

### 코드 스타일 및 안전성

**모듈화**
기능별 파일 분리(`export`/`import`)를 원칙으로 한다.

**포맷**
들여쓰기는 Space 4칸, 문장 끝에는 반드시 세미콜론(`;`)을 사용한다.

**안전성**
`null`/`undefined` 체크를 철저히 하며, Optional Chaining(`?.`)을 적극 활용한다.

**주석**
복잡한 로직에는 동작 원리를 설명하는 한국어 주석을 달고, 함수 헤더에 역할과 매개변수 설명을 명시한다.

## 4. 금기 사항 및 자동 과업

### 금기 사항

- **매직 넘버 하드코딩 금지**: 모든 상수는 프로젝트의 전용 상수 파일로 분리한다. 파일명과 경로는 `SPEC.md`에서 프로젝트별로 정의한다. 미정의 시 기본값은 `Config.js`로 한다.
- **전역 변수 사용 지양**: 클래스 멤버나 모듈 스코프를 활용한다.
- **주석 내 영문 혼용 금지**: 고유 전문 용어(라이브러리명, 표준 명칭 등)는 예외로 한다.

### 자동 문서화

작업 완료 직후, 별도 요청이 없어도 다음 문서를 최신 상태로 갱신한다.

- `.ai/STORY.md`: 상세 변경 이력 기록.
- `.ai/SPEC.md`: 기능 명세 및 구조 현행화.

---

**변경 요약 (원본 대비)**
1. 사전 참조 파일 부재 시 처리 절차 추가
2. 지시 우선순위 계층 명시
3. Review 단계 출력 범위 정의
4. Verify 단계 신설 (조건부 적용)
5. Config.js 고정 참조 → 전용 상수 파일로 추상화, SPEC.md 위임 구조로 전환

계획 승인한다. 단, 아래 규칙 반드시 지켜:
1. 파일 하나 완성할 때마다 알려주고 내 승인 받을 것
2. 에러 나면 혼자 수정 시도하지 말고 에러 메시지 그대로 나한테 보고
3. 다음 파일로 넘어가기 전에 반드시 "다음은 [파일명] 작성할게요, 진행할까요?" 확인
