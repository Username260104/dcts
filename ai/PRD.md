# PRD: 디자인 커뮤니케이션 번역 시스템 MVP

## 문서 정보

- **프로젝트명**: Design Communication Translation System (DCTS)
- **버전**: MVP v1.0
- **작성 기준**: 온톨로지 v2 (`ontology_v2.xlsx`)
- **현재 단계**: Week 3 — 프론트엔드 기본 구조 & 질문 엔진 구현

---

## 1. 프로젝트 개요

### 1.1 한 줄 정의

브랜드 디자인 에이전시의 **피드백 단계**에서, 클라이언트의 모호한 수정 피드백("좀 더 고급스럽게")을 산파법 질문을 통해 구체화하고, 디자이너 팀 전원이 같은 방향을 보고 작업할 수 있는 "수정 방향 브리프"를 자동 생성하는 웹앱.

### 1.2 핵심 문제

1. **클라이언트의 문제**: 머릿속에 분명히 그림이 있지만, 디자인 어휘와 배경지식의 한계로 "좀 더 고급스럽게" 수준으로밖에 전달하지 못한다.
2. **디자이너의 문제**: 같은 피드백을 받아도 디자이너마다 머릿속에 그리는 그림이 다르다. 언어→비주얼 번역이 내재화/통일되지 않아 리소스와 시간이 소모된다.

### 1.3 해결 방식

- **산파법(Socratic Method)**: 클라이언트에게 직접 "뭘 원하세요?"라고 묻지 않는다. 대신 선택지와 시각 비교를 통해 머릿속 그림을 단계적으로 꺼낸다.
- **Progressive Disclosure**: 한 번에 하나의 질문만. 이전 답변이 다음 질문의 방향을 결정한다.
- **이중 출력**: 같은 해석 결과를 클라이언트용(비전문가 언어 확인)과 디자이너용(전문 토큰 + 조정 방향)으로 각각 출력한다.

### 1.4 사용 시점

이 시스템은 **최초 인터뷰 도구가 아니다**. 에이전시가 1차 시안을 제출한 후, 클라이언트로부터 수정 피드백이 돌아왔을 때 사용한다.

```
1차 시안 제출 → 클라이언트 피드백 ("좀 더 고급스럽게")
→ AE가 클라이언트에게 이 도구 링크 전달
→ 클라이언트가 직접 질문에 답변
→ 수정 방향 브리프 생성
→ 디자이너 팀 전원이 브리프를 보고 같은 방향으로 수정 작업
```

### 1.5 사용자 정의

| 역할 | 시스템 내 행동 | 핵심 니즈 |
|------|-------------|---------|
| **클라이언트** (주 사용자) | 피드백 입력, 질문 응답, 결과 확인 | 디자인 몰라도 쉽게 답할 수 있어야 함. 3~5분 안에 끝나야 함 |
| **AE/PM** | 링크 전달, 맥락 정보 사전 입력 가능 | 클라이언트에게 부담 없이 전달할 수 있어야 함 |
| **디자이너** (결과 소비자) | 브리프 열람, 공유 | 브리프만 보고 바로 작업 가능해야 함. 해석의 여지가 최소화 |

## 2. 데이터 구조 — 온톨로지 v2

온톨로지 v2 (`ontology_v2.xlsx`)가 이 시스템의 핵심 데이터이다. 5개 시트로 구성되어 있으며, 이를 JSON으로 변환하여 앱에서 사용한다.

### 2.1 시트 1: 피드백 트리거 사전

클라이언트가 실제로 쓰는 피드백 표현 25개. 시스템의 진입점.

```json
{
  "id": 1,
  "expression": "좀 더 고급스럽게 해주세요",
  "type": "directional",       // "directional" | "negative" | "ambiguous"
  "axis": "품격/가치",
  "candidate_branches": ["LUX-A", "LUX-B", "LUX-C", "LUX-D"],
  "co_expressions": ["근데 너무 올드하진 않게", "세련되게"],
  "risk_level": 5,             // 1~5, 해석 위험도
  "entry_question": "고급스러움에도 여러 결이 있는데요, 지금 시안에서 어떤 느낌을 더 살리고 싶으세요?"
}
```

**표현 유형별 처리 전략**:
- `directional` (방향형, 12개): 해당 형용사의 분기 세분화 질문으로 진입
- `negative` (부정형, 8개): 현재 시안의 어떤 요소가 문제인지 특정하는 질문으로 진입
- `ambiguous` (모호형, 5개): 시각 비교(A/B)로 진입

### 2.2 시트 2: 해석 분기 + 안티 패턴

37개 분기. 12개 감성 형용사 × 3~4개 분기. 이 시스템의 핵심 해석 레이어.

```json
{
  "adjective": "고급스러운",
  "branch_id": "LUX-B",
  "branch_label": "미니멀 럭셔리",
  "description_designer": "절제된 고급감, Less is More, 조용한 사치. 여백 자체가 가치",
  "description_client": "비싼 호텔 로비처럼, 아무것도 없는데 고급스러운 느낌",
  "confusable_branches": ["CLN-A", "SOP-C", "MOD-A"],
  "distinction_key": "CLN-A(스위스 미니멀)는 기능적 정리, LUX-B는 의도된 여유. 비운 이유가 다름",
  "contrast_question": "불필요한 걸 치워서 깔끔한 건가요, 일부러 비워둬서 여유로운 건가요?",
  "anti_signals": ["알차게", "정보가 많았으면", "화려하게", "눈에 확 띄게"],
  "excluded_tokens": ["높은 정보밀도", "그래디언트", "형광 강조색", "모노스페이스", "대시보드 레이아웃", "장식적 서체"],
  "references": ["Celine", "Aesop", "Bottega Veneta"]
}
```

**핵심 필드 설명**:
- `contrast_question`: 인접 분기와 구분하는 클라이언트용 질문. **반드시 비전문가 언어**. 항상 "지금 시안 대비" 프레임.
- `anti_signals`: 클라이언트가 이 표현을 쓰면 이 분기가 아니라는 신호. 질문 엔진에서 분기 후보 제거에 사용.
- `confusable_branches`: 디자이너가 혼동하기 쉬운 인접 분기. 브리프 출력 시 "흔한 실수 경고"에 사용.

### 2.3 시트 3: 디자인 토큰 (조정 방향형)

37개 분기 각각에 대한 디자인 조정 방향. **절대 토큰이 아니라 "현재 시안에서 이 방향으로 가려면 뭘 건드려야 하는지"**.

```json
{
  "branch_id": "LUX-B",
  "color_direction": "팔레트 극도로 축소 (2~3색). 뮤트/뉴트럴로 이동. 채도 대폭 낮추기...",
  "typography_direction": "산세리프로 전환. 웨이트 얇게 (Light~Regular). 자간 넓히기...",
  "layout_direction": "여백 비율 대폭 확대 (60% 이상). 요소 수 절반 이하로 줄이기...",
  "image_direction": "하이키 라이팅. 자연광. 배경 단순화. 오브제 수 최소화...",
  "texture_direction": "매트/무광. 콘크리트, 면, 종이 질감...",
  "never_do": "높은 정보밀도 금지. 그래디언트 금지. 장식적 서체 금지..."
}
```

### 2.4 시트 4: 맥락 변수

질문 플로우 진입 전 수집하는 브랜드 컨텍스트. 분기 후보군의 우선순위에 가중치를 줌.

```json
{
  "industry": {
    "뷰티/코스메틱": { "boost": ["LUX-B", "NAT-A", "SOP-A", "SOP-B"], "suppress": [] },
    "테크/SaaS": { "boost": ["MOD-A", "TRS-B", "SOP-C", "CLN-C"], "suppress": [] },
    ...
  },
  "price_position": {
    "럭셔리": { "boost": ["LUX-A", "LUX-B", "LUX-C", "LUX-D", "SOP-A"], "suppress": ["VIV-C", "CMF-B"] },
    ...
  },
  "project_stage": {
    "1차 시안": { "question_breadth": "wide" },
    "2차 수정": { "question_breadth": "narrow" },
    "3차 이상": { "question_breadth": "micro", "warn_direction_change": true }
  },
  "target_age": { ... }
}
```

### 2.5 시트 5: 수정 방향 브리프 템플릿

최종 출력물의 구조. 두 겹 (클라이언트 확인용 + 디자이너용).

```
A. 원문 피드백 — 클라이언트 원문 그대로
B. 해석 요약 (클라이언트 확인용) — 비전문가 언어로 "맞으신가요?" 확인
C. 해석 요약 (디자이너용) — 확정 분기, 배제 분기, 해석 근거
D. 핵심 조정 포인트 — 컬러/서체/레이아웃/이미지/질감 각각의 조정 방향
E. 절대 하지 말 것 — 배제 토큰 + 흔한 실수 경고
F. 레퍼런스 방향 — 참조 브랜드 + 안티 레퍼런스
```

## 3. 화면 구조 & 유저 플로우

### 3.1 전체 플로우

```
[진입 화면] → [맥락 입력] → [질문 플로우] → [클라이언트 확인] → [브리프 출력]
    (1)          (2)           (3)              (4)               (5)
```

### 3.2 화면 1: 진입 화면

**목적**: 클라이언트가 피드백을 입력하는 진입점.

**구성 요소**:
- 헤딩: "디자인 피드백을 더 정확하게 전달해보세요" (또는 에이전시가 커스터마이징)
- 서브텍스트: "몇 가지 질문에 답하시면, 원하시는 방향을 디자이너에게 정확히 전달해드릴게요."
- 텍스트 입력 필드: "어떤 피드백을 전달하고 싶으세요?" (placeholder: "예: 좀 더 고급스럽게 해주세요")
- 또는 빈출 트리거 표현을 선택지로 제시 (칩/태그 형태)

**트리거 매칭 로직 (MVP)**:
- 클라이언트 입력 텍스트를 트리거 사전(시트 1)과 키워드 매칭
- 정확 매칭 실패 시 → 부분 매칭 (형용사 추출) 시도
- 매칭 실패 시 → 모호형(`ambiguous`)으로 처리, A/B 비교 플로우로 진입
- **MVP에서는 자연어 처리 없이 키워드 매칭 + 선택지 fallback으로 충분**

**UX 주의사항**:
- 입력이 부담스러우면 안 됨. 빈출 표현 칩을 먼저 보여주고, 없으면 직접 입력하게.
- 톤은 친근하고 비강압적. "정확히 안 적어도 괜찮아요, 같이 찾아볼게요" 느낌.

### 3.3 화면 2: 맥락 입력

**목적**: 분기 가중치 결정을 위한 최소한의 브랜드 컨텍스트 수집.

**구성 요소** (3문항, 각각 선택지 형태):
1. "어떤 분야의 브랜드인가요?" — 산업군 선택 (시트 4 기준, 10개)
2. "가격대는 어떤 포지션인가요?" — 럭셔리/프리미엄/미들/매스 (4개)
3. "몇 번째 수정인가요?" — 1차/2차/3차 이상 (3개)

**UX 주의사항**:
- AE가 사전에 입력해두고 클라이언트에게 링크를 보내는 시나리오도 지원해야 함 (URL 파라미터로 맥락 정보 프리셋 가능하게)
- "잘 모르겠어요" 선택지도 제공 (기본값 적용)
- 이 화면은 빠르게 넘겨야 함. 진짜 시작은 다음 질문 플로우.

### 3.4 화면 3: 질문 플로우 (핵심)

**목적**: 산파법으로 클라이언트의 의도를 구체화하는 단계.

**기본 원칙**:
- 한 번에 하나의 질문만 보여준다 (Progressive Disclosure)
- 모든 질문은 "지금 시안 대비" 프레임
- 전문 용어 절대 사용 금지. 클라이언트가 디자인을 몰라도 답할 수 있어야 함
- 총 질문 수는 **3~5개 이내**. 5개를 넘기면 안 됨
- 매 질문마다 진행률 표시 (예: 3/5)

**질문 유형**:

1. **텍스트 선택형**: 대비 질문 2개 선택지 + "둘 다 아닌데요" 옵션
   ```
   "묵직하고 전통 있는 느낌을 원하시나요,
    비워내서 여유로운 느낌을 원하시나요?"
   
   [묵직한 전통] [여유로운 비움] [둘 다 아닌데요]
   ```

2. **이미지 A/B 비교형**: 두 분기의 레퍼런스 이미지를 나란히 보여주고 선택
   ```
   "어느 쪽이 더 원하시는 방향에 가까우세요?"
   
   [이미지 A]  [이미지 B]  [둘 다 아닌데요]
   ```

3. **부정 확인형** (부정형 트리거 진입 시): 현재 시안의 문제 요소를 특정
   ```
   "딱딱하다고 느끼신 부분이 어디인가요?"
   
   [색감이 차가워요] [글씨가 딱딱해요] [전체 분위기가요] [잘 모르겠어요]
   ```

**"둘 다 아닌데요" 처리** (탈출 경로):
- 현재 대비 축과 다른 축의 질문으로 전환
- 2번 연속 "둘 다 아닌데요" 선택 시 → 이미지 A/B 비교로 전환
- 3번 연속 시 → "직접 설명해주셔도 좋아요" 자유 텍스트 입력 제공

**질문 엔진 로직**:

```
상태:
  - remaining_candidates: [분기 ID 배열]  // 아직 남은 후보 분기들
  - eliminated: [분기 ID 배열]           // 배제된 분기들
  - answer_history: [{ question, answer, eliminated_branches }]
  - confidence: number                   // 0~1, 수렴 확신도

초기 상태:
  - 트리거 매칭 결과의 candidate_branches로 시작
  - 맥락 변수에 따라 후보 순위 정렬 (boost/suppress)

매 질문 후:
  1. 선택된 방향과 반대되는 분기들을 remaining_candidates에서 제거
  2. anti_signals에 해당하는 분기 제거
  3. 남은 후보 중 가장 구분이 필요한 쌍의 contrast_question을 다음 질문으로 선택

수렴 조건 (질문 종료):
  - remaining_candidates가 2개 이하로 줄었을 때
  - 또는 연속 2회 같은 방향 선택 시
  - 또는 총 질문 5개 도달 시 (강제 수렴)

수렴 시 출력:
  - primary_branch: 최종 1순위 분기
  - secondary_branch: 2순위 참조 분기 (있을 경우)
  - eliminated_branches: 명시적으로 배제된 분기들
  - answer_log: 전체 질문-응답 이력
```

### 3.5 화면 4: 클라이언트 확인

**목적**: "이렇게 이해했는데 맞으세요?" 확인 단계.

**구성 요소**:
- 해석 요약 (클라이언트용 언어): "말씀하신 '고급스러움'은 전통적 무게감보다는 비워내는 여유에 가까운 것 같아요."
- "이건 아닌" 방향도 함께 표시: "전통적이고 묵직한 고급스러움은 아닌 것 같아요."
- 레퍼런스 이미지 2~3장 (확정 분기의 대표 이미지)
- 안티 레퍼런스 1~2장 ("이런 느낌은 아닌 거죠?")
- [맞아요, 이 방향이에요] / [아니에요, 다시 해볼게요] 버튼

**"아니에요" 선택 시**:
- 질문 플로우로 돌아가되, 이전 답변 이력은 유지
- 추가 질문 1~2개로 보정

### 3.6 화면 5: 수정 방향 브리프 출력

**목적**: 디자이너 팀이 같은 방향을 보고 작업할 수 있는 최종 산출물.

**구성 요소** (시트 5 템플릿 기반):

**A. 헤더**
- 원문 피드백 (클라이언트가 입력한 그대로)
- 맥락 정보 (산업군 / 가격대 / 수정 차수)
- 생성 일시

**B. 해석 요약**
- 확정 분기 라벨 + 설명 (디자이너용)
- 배제 분기 라벨 + 이유
- 해석 근거 (질문-응답 로그 핵심 요약)

**C. 핵심 조정 포인트**
- 컬러 조정 방향
- 서체 조정 방향
- 레이아웃 조정 방향
- 이미지/사진 조정 방향
- 질감/텍스처 조정 방향

**D. 절대 하지 말 것**
- 배제 토큰 리스트
- 흔한 실수 경고 (혼동 분기와의 차이 강조)

**E. 레퍼런스**
- 참조 이미지 (분기 레퍼런스 풀에서)
- 안티 레퍼런스 이미지 ("이건 아닙니다")

**출력 형태**:
- 웹 페이지로 바로 열람 (고유 URL)
- PDF 다운로드
- 링크 복사 (디자이너 팀 공유용)

## 4. 기술 요구사항

### 4.1 기술 스택

- **프론트엔드**: React (Next.js 권장) + TypeScript
- **스타일링**: Tailwind CSS
- **상태관리**: React Context 또는 Zustand (질문 엔진 상태)
- **데이터**: 온톨로지 JSON 파일 (static import, DB 불필요 — MVP)
- **PDF 생성**: html2pdf 또는 react-pdf (브리프 다운로드용)
- **배포**: Vercel

### 4.2 데이터 흐름

```
ontology_v2.xlsx
  → 빌드 타임에 JSON 변환 (수동 또는 스크립트)
    → triggers.json      (시트 1)
    → branches.json      (시트 2)
    → tokens.json        (시트 3)
    → context.json       (시트 4)
    → brief_template.json (시트 5)

런타임:
  사용자 입력 → 트리거 매칭 → 질문 엔진 (상태 머신) → 브리프 생성 → 출력
```

### 4.3 핵심 모듈

**1. TriggerMatcher**
```
입력: 클라이언트 텍스트
출력: { matched_trigger, type, candidate_branches }
방식: 키워드 매칭 + fallback (빈출 표현 선택지 제시)
```

**2. QuestionEngine (상태 머신)**
```
상태: { remaining_candidates, eliminated, answer_history, confidence, question_count }
메서드:
  - getNextQuestion(): 남은 후보 중 가장 구분 필요한 쌍의 contrast_question 반환
  - submitAnswer(answer): 후보 분기 갱신, 배제 분기 업데이트
  - isConverged(): 수렴 여부 판정
  - getResult(): 최종 확정 분기 + 배제 분기 + 응답 로그
```

**3. BriefGenerator**
```
입력: { primary_branch, secondary_branch, eliminated_branches, answer_log, context }
출력: { client_summary, designer_brief, adjustment_tokens, never_do, references }
방식: 시트 2~5 데이터 조합 + 템플릿 렌더링
```

### 4.4 URL 구조

```
/ — 진입 화면
/session/:id — 질문 플로우 (세션별 고유 ID)
/session/:id/confirm — 클라이언트 확인 화면
/brief/:id — 브리프 열람 (디자이너 공유용 고유 URL)
/brief/:id/pdf — PDF 다운로드
```

### 4.5 맥락 정보 프리셋 (AE용)

AE가 클라이언트에게 링크 보낼 때 맥락 정보를 미리 채워서 보낼 수 있어야 함.

```
/session/new?industry=beauty&price=premium&stage=1st
```

## 5. 레퍼런스 이미지 관련

레퍼런스 이미지 풀 구축은 **별도 병렬 작업**으로 진행 중이다. Week 3 구현 시점에는 이미지가 완비되지 않을 수 있다.

### 5.1 MVP 대응 전략

- 이미지가 없는 분기는 **텍스트 선택형 질문만 사용** (이미지 A/B 비교 건너뜀)
- 이미지가 있는 분기부터 이미지 비교 활성화
- 브리프 출력의 레퍼런스 섹션도 이미지 유무에 따라 동적 처리

### 5.2 이미지 데이터 구조

```json
{
  "branch_id": "LUX-B",
  "reference_images": [
    { "url": "/images/ref/LUX-B-01.jpg", "caption": "Celine 2024 캠페인" },
    { "url": "/images/ref/LUX-B-02.jpg", "caption": "Aesop 매장 인테리어" },
    { "url": "/images/ref/LUX-B-03.jpg", "caption": "Bottega Veneta 패키지" }
  ],
  "anti_reference_images": [
    { "url": "/images/anti/LUX-B-anti-01.jpg", "caption": "대시보드 UI (CLN-A 혼동 주의)" },
    { "url": "/images/anti/LUX-B-anti-02.jpg", "caption": "장식 과잉 (LUX-D 혼동 주의)" }
  ]
}
```

### 5.3 이미지 없을 때 fallback

이미지 A/B 비교가 불가능한 경우, 해당 질문을 텍스트 대비 질문으로 대체:
```
이미지 있을 때: [이미지 A] vs [이미지 B]
이미지 없을 때: "비워내서 여유로운 느낌" vs "정확하게 맞춘 깔끔한 느낌"
                 (description_client 필드 활용)
```

## 6. 디자인 가이드라인

### 6.1 톤 & 무드

이 앱의 톤은 **"잘 듣는 통역사"**. 권위적이지 않고, 클라이언트의 말을 존중하면서 부드럽게 명확화해주는 느낌.

- 질문은 항상 존댓말, 부드러운 어조
- "틀린 답은 없어요", "같이 찾아볼게요" 같은 안심 문구
- 디자인 용어는 클라이언트 화면에서 절대 노출하지 않음
- 디자이너 브리프에서만 전문 용어 사용

### 6.2 UI 원칙

- **한 화면에 하나의 행동만**. 질문 하나, 선택지 2~3개, 다음 버튼.
- **진행 감각**: 상단에 프로그레스 바 또는 스텝 인디케이터
- **되돌아가기 가능**: 이전 질문으로 돌아가서 답변 수정 가능
- **모바일 퍼스트**: 클라이언트는 모바일에서 볼 가능성 높음
- **이미지 A/B 비교**: 두 이미지를 나란히 (데스크톱) 또는 상하로 (모바일) 배치. 탭하면 선택.

### 6.3 시각적 방향

- 이 앱 자체가 특정 디자인 스타일을 강하게 가지면 안 됨 (레퍼런스 이미지 판단에 영향)
- 뉴트럴하고 깨끗한 UI. 콘텐츠(질문, 이미지)가 주인공.
- 배경 화이트/라이트그레이, 텍스트 다크, 강조색 1개 (브랜드 컬러 또는 뉴트럴 블루)

## 7. MVP 스코프 경계

### 7.1 MVP에 포함

- 피드백 텍스트 입력 + 트리거 키워드 매칭

- 맥락 정보 3문항 입력

- 텍스트 선택형 질문
- 이미지 A/B 비교 (이미지 있는 분기만)
- "둘 다 아닌데요" 탈출 경로
- 클라이언트 확인 화면
- 디자이너용 수정 방향 브리프 자동 생성
- 브리프 웹 열람 + 링크 공유
- PDF 다운로드
- URL 파라미터로 맥락 프리셋
- 모바일 반응형

### 7.2 MVP에 미포함 (Phase 2)

- 자연어 처리 (LLM 기반 트리거 매칭)
- AI 동적 질문 생성 (정적 트리 → 동적)

- 멀티모달 입력 (클라이언트가 이미지 드롭)
- 분기 가중합 출력 (LUX-B 45% + SOP-B 25%)
- 사용자 계정 / 로그인
- 프로젝트 히스토리 대시보드
- Slack/노션 연동
- 피드백 루프 (최종 디자인 역분석 → 온톨로지 보정)
- 수치 벡터 좌표화 (8축 1~7점)
- 에이전시별 온톨로지 커스터마이징

## 8. 성공 지표

### 8.1 핵심 측정 (파일럿 시)

| 지표 | 측정 방법 | 목표 |
|-----|---------|-----|
| 수정 횟수 감소 | 시스템 사용 프로젝트 vs 미사용 프로젝트 | 30% 이상 감소 |
| 클라이언트 완주율 | 진입 → 브리프 확인까지 도달 비율 | 70% 이상 |
| 디자이너 방향 일치율 | 브리프만 보고 2명 이상 같은 방향 제시 | 80% 이상 |
| 클라이언트 소요 시간 | 진입 → 확인까지 | 3~5분 이내 |
| 클라이언트 "맞아요" 비율 | 확인 화면에서 1회 통과 | 75% 이상 |

### 8.2 관찰 지표

- 어떤 트리거 표현이 가장 빈출하는지
- 어떤 질문에서 "둘 다 아닌데요"가 많이 나오는지 (질문 품질 문제)
- 어떤 분기 쌍이 가장 자주 혼동되는지
- 어떤 산업군에서 가장 자주 사용되는지

## 9. 주의사항 & 핵심 리스크

### 9.1 가장 중요한 원칙

**이 시스템의 주 목적은 클라이언트의 머릿속에 있는 그림을 산파해내는 것이다.** 디자이너 편의는 부수 효과. 모든 설계 판단에서 "클라이언트가 이 질문에 쉽게 답할 수 있는가?"가 최우선 기준.

### 9.2 핵심 리스크

1. **클라이언트 이탈**: 질문이 5개를 넘거나, 질문이 어렵거나, 가치를 못 느끼면 이탈함. 매 질문이 "아, 그래 내가 원한 게 이거야!"라는 발견의 순간을 줘야 함.

2. **트리거 매칭 실패**: 클라이언트가 트리거 사전에 없는 표현을 쓸 때. MVP에서는 빈출 표현 선택지 fallback으로 대응하되, 매칭 실패 로그를 수집해서 사전 확장에 활용.

3. **브리프를 디자이너가 안 봄**: 기존 워크플로우에 끼어들지 못하면 쓰이지 않음. MVP에서는 링크 공유 + PDF로 최소한 전달 가능하게, Phase 2에서 Slack/노션 연동.

4. **온톨로지 품질**: 시스템의 가치는 코드가 아니라 데이터 품질에 달려있음. 잘못된 대비 질문, 부정확한 토큰 매핑은 코드로 커버 불가. Week 5 테스트에서 반드시 검증.
