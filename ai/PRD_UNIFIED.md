# PRD: 디자인 커뮤니케이션 번역 시스템 (DCTS)

## 문서 정보

- **프로젝트명**: Design Communication Translation System (DCTS)
- **버전**: v1.0
- **작성 기준**: 온톨로지 v2 (`ontology_v2.xlsx`)
- **구현 단계**: 2-Phase 구조
  - **Phase 1**: 정적 트리 기반 MVP (현재 구현 대상)
  - **Phase 2**: LLM 오케스트레이션 통합 (Phase 1 완료 후)

---

## 1. 프로젝트 개요

### 1.1 한 줄 정의

브랜드 디자인 에이전시의 **피드백 단계**에서, 클라이언트의 모호한 수정 피드백("좀 더 고급스럽게")을 산파법 질문을 통해 구체화하고, 디자이너 팀 전원이 같은 방향을 보고 작업할 수 있는 "수정 방향 브리프"를 자동 생성하는 웹앱.

### 1.2 핵심 문제

1. **클라이언트의 문제**: 머릿속에 분명히 그림이 있지만, 디자인 어휘와 배경지식의 한계로 "좀 더 고급스럽게" 수준으로밖에 전달하지 못한다.
2. **디자이너의 문제**: 같은 피드백을 받아도 디자이너마다 머릿속에 그리는 그림이 다르다. 언어→비주얼 번역이 내재화/통일되지 않아 리소스와 시간이 소모된다.

### 1.3 해결 방식

- **산파법(Socratic Method)**: 클라이언트에게 직접 "뭘 원하세요?"라고 묻지 않는다. 대신 선택지와 시각 비교를 통해 머릿속 그림을 단계적으로 꺼낸다.
- **Progressive Disclosure**: 한 번에 하나의 질문만. 이전 답변이 다음 질문의 방향을 결정한다.
- **이중 출력**: 같은 해석 결과를 클라이언트용(비전문가 언어 확인)과 디자이너용(전문 토큰 + 조정 방향)으로 각각 출력한다.
- **LLM 오케스트레이션** (Phase 2): 온톨로지를 컨텍스트로 갖고, 매 응답마다 후보 분기의 확정/미확정 영역을 판단하여 가장 효율적인 다음 질문을 동적으로 생성한다.

### 1.4 사용 시점

이 시스템은 **최초 인터뷰 도구가 아니다**. 에이전시가 1차 시안을 제출한 후, 클라이언트로부터 수정 피드백이 돌아왔을 때 사용한다.

```
1차 시안 제출 → 클라이언트 피드백 ("좀 더 고급스럽게")
→ AE가 클라이언트에게 이 도구 링크 전달
→ 클라이언트가 직접 질문에 답변
→ 수정 방향 브리프 생성
→ 디자이너 팀 전원이 브리프를 보고 같은 방향으로 수정 작업
```

### 1.5 사용자 정의

| 역할 | 시스템 내 행동 | 핵심 니즈 |
|------|-------------|---------|
| **클라이언트** (주 사용자) | 피드백 입력, 질문 응답, 결과 확인 | 디자인 몰라도 쉽게 답할 수 있어야 함. 3~5분 안에 끝나야 함 |
| **AE/PM** | 링크 전달, 맥락 정보 사전 입력 가능 | 클라이언트에게 부담 없이 전달할 수 있어야 함 |
| **디자이너** (결과 소비자) | 브리프 열람, 공유 | 브리프만 보고 바로 작업 가능해야 함. 해석의 여지가 최소화 |

### 1.6 2-Phase 전략

이 시스템은 두 단계로 구현한다.

**Phase 1 (정적 트리 MVP)**: 온톨로지 데이터 기반의 정적 의사결정 트리로 질문 엔진을 구현한다. 키워드 매칭 + 조건부 분기로 동작. 핵심 가치("클라이언트 어휘 확장 → 디자이너 의식 통일")가 정적 트리만으로도 성립하는지 검증한다.

**Phase 2 (LLM 오케스트레이션)**: Phase 1의 정적 트리를 LLM 기반 동적 질문 엔진으로 교체한다. 자연어 이해, 교차 감성 처리, 맥락 누적 해석이 가능해진다. Phase 1의 정적 트리 코드는 LLM 장애 시 fallback으로 유지한다.

## 2. 데이터 구조 — 온톨로지 v2

온톨로지 v2 (`ontology_v2.xlsx`)가 이 시스템의 핵심 데이터이다. 5개 시트로 구성되어 있으며, 이를 JSON으로 변환하여 앱에서 사용한다. Phase 1에서는 정적 데이터로, Phase 2에서는 LLM 컨텍스트로 활용된다.

### 2.1 시트 1: 피드백 트리거 사전

클라이언트가 실제로 쓰는 피드백 표현 25개. 시스템의 진입점.

```json
{
  "id": 1,
  "expression": "좀 더 고급스럽게 해주세요",
  "type": "directional",       // "directional" | "negative" | "ambiguous"
  "axis": "품격/가치",
  "candidate_branches": ["LUX-A", "LUX-B", "LUX-C", "LUX-D"],
  "co_expressions": ["근데 너무 올드하진 않게", "세련되게"],
  "risk_level": 5,             // 1~5, 해석 위험도
  "entry_question": "고급스러움에도 여러 결이 있는데요, 지금 시안에서 어떤 느낌을 더 살리고 싶으세요?"
}
```

**표현 유형별 처리 전략**:
- `directional` (방향형, 12개): 해당 형용사의 분기 세분화 질문으로 진입
- `negative` (부정형, 8개): 현재 시안의 어떤 요소가 문제인지 특정하는 질문으로 진입
- `ambiguous` (모호형, 5개): 시각 비교(A/B)로 진입

### 2.2 시트 2: 해석 분기 + 안티 패턴

37개 분기. 12개 감성 형용사 × 3~4개 분기. 이 시스템의 핵심 해석 레이어.

```json
{
  "adjective": "고급스러운",
  "branch_id": "LUX-B",
  "branch_label": "미니멀 럭셔리",
  "description_designer": "절제된 고급감, Less is More, 조용한 사치. 여백 자체가 가치",
  "description_client": "비싼 호텔 로비처럼, 아무것도 없는데 고급스러운 느낌",
  "confusable_branches": ["CLN-A", "SOP-C", "MOD-A"],
  "distinction_key": "CLN-A(스위스 미니멀)는 기능적 정리, LUX-B는 의도된 여유. 비운 이유가 다름",
  "contrast_question": "불필요한 걸 치워서 깔끔한 건가요, 일부러 비워둬서 여유로운 건가요?",
  "anti_signals": ["알차게", "정보가 많았으면", "화려하게", "눈에 확 띄게"],
  "excluded_tokens": ["높은 정보밀도", "그래디언트", "형광 강조색", "모노스페이스", "대시보드 레이아웃", "장식적 서체"],
  "references": ["Celine", "Aesop", "Bottega Veneta"]
}
```

**핵심 필드 설명**:
- `contrast_question`: 인접 분기와 구분하는 클라이언트용 질문. **반드시 비전문가 언어**. 항상 "지금 시안 대비" 프레임. Phase 1에서는 정적으로 호출, Phase 2에서는 LLM이 맥락에 맞게 변형하되 이 데이터를 참조.
- `anti_signals`: 클라이언트가 이 표현을 쓰면 이 분기가 아니라는 신호. 질문 엔진에서 분기 후보 제거에 사용.
- `confusable_branches`: 디자이너가 혼동하기 쉬운 인접 분기. 브리프 출력 시 "흔한 실수 경고"에 사용.

### 2.3 시트 3: 디자인 토큰 (조정 방향형)

37개 분기 각각에 대한 디자인 조정 방향. **절대 토큰이 아니라 "현재 시안에서 이 방향으로 가려면 뭘 건드려야 하는지"**.

```json
{
  "branch_id": "LUX-B",
  "color_direction": "팔레트 극도로 축소 (2~3색). 뮤트/뉴트럴로 이동. 채도 대폭 낮추기...",
  "typography_direction": "산세리프로 전환. 웨이트 얇게 (Light~Regular). 자간 넓히기...",
  "layout_direction": "여백 비율 대폭 확대 (60% 이상). 요소 수 절반 이하로 줄이기...",
  "image_direction": "하이키 라이팅. 자연광. 배경 단순화. 오브제 수 최소화...",
  "texture_direction": "매트/무광. 콘크리트, 면, 종이 질감...",
  "never_do": "높은 정보밀도 금지. 그래디언트 금지. 장식적 서체 금지..."
}
```

### 2.4 시트 4: 맥락 변수

질문 플로우 진입 전 수집하는 브랜드 컨텍스트. 분기 후보군의 우선순위에 가중치를 줌.

```json
{
  "industry": {
    "뷰티/코스메틱": { "boost": ["LUX-B", "NAT-A", "SOP-A", "SOP-B"], "suppress": [] },
    "테크/SaaS": { "boost": ["MOD-A", "TRS-B", "SOP-C", "CLN-C"], "suppress": [] }
  },
  "price_position": {
    "럭셔리": { "boost": ["LUX-A", "LUX-B", "LUX-C", "LUX-D", "SOP-A"], "suppress": ["VIV-C", "CMF-B"] }
  },
  "project_stage": {
    "1차 시안": { "question_breadth": "wide" },
    "2차 수정": { "question_breadth": "narrow" },
    "3차 이상": { "question_breadth": "micro", "warn_direction_change": true }
  },
  "target_age": { ... }
}
```

### 2.5 시트 5: 수정 방향 브리프 템플릿

최종 출력물의 구조. 두 겹 (클라이언트 확인용 + 디자이너용).

```
A. 원문 피드백 — 클라이언트 원문 그대로
B. 해석 요약 (클라이언트 확인용) — 비전문가 언어로 "맞으신가요?" 확인
C. 해석 요약 (디자이너용) — 확정 분기, 배제 분기, 해석 근거
D. 핵심 조정 포인트 — 컬러/서체/레이아웃/이미지/질감 각각의 조정 방향
E. 절대 하지 말 것 — 배제 토큰 + 흔한 실수 경고
F. 레퍼런스 방향 — 참조 브랜드 + 안티 레퍼런스
```

## 3. 화면 구조 & 유저 플로우

화면 구조는 Phase 1, Phase 2 모두 동일하다. 바뀌는 건 질문 생성과 분기 판정의 백엔드 로직뿐.

### 3.1 전체 플로우

```
[진입 화면] → [맥락 입력] → [질문 플로우] → [클라이언트 확인] → [브리프 출력]
    (1)          (2)           (3)              (4)               (5)
```

### 3.2 화면 1: 진입 화면

**목적**: 클라이언트가 피드백을 입력하는 진입점.

**구성 요소**:
- 헤딩: "디자인 피드백을 더 정확하게 전달해보세요" (또는 에이전시가 커스터마이징)
- 서브텍스트: "몇 가지 질문에 답하시면, 원하시는 방향을 디자이너에게 정확히 전달해드릴게요."
- 텍스트 입력 필드: "어떤 피드백을 전달하고 싶으세요?" (placeholder: "예: 좀 더 고급스럽게 해주세요")
- 또는 빈출 트리거 표현을 선택지로 제시 (칩/태그 형태)

**트리거 매칭 로직**:
- **Phase 1**: 키워드 매칭. 클라이언트 입력 텍스트를 트리거 사전(시트 1)과 매칭. 정확 매칭 실패 시 부분 매칭(형용사 추출) 시도. 매칭 실패 시 모호형(`ambiguous`)으로 처리, A/B 비교 플로우로 진입.
- **Phase 2**: LLM 자연어 이해. 트리거 사전에 없는 표현("뭔가 좀 후지게 느껴져요", "경쟁사 A처럼은 안 됐으면")도 처리 가능. 온톨로지 전체를 컨텍스트로 보고 피드백 유형 판정 + 초기 후보 분기 식별.

**UX 주의사항**:
- 입력이 부담스러우면 안 됨. 빈출 표현 칩을 먼저 보여주고, 없으면 직접 입력하게.
- 톤은 친근하고 비강압적. "정확히 안 적어도 괜찮아요, 같이 찾아볼게요" 느낌.

### 3.3 화면 2: 맥락 입력

**목적**: 분기 가중치 결정을 위한 최소한의 브랜드 컨텍스트 수집.

**구성 요소** (3문항, 각각 선택지 형태):
1. "어떤 분야의 브랜드인가요?" — 산업군 선택 (시트 4 기준, 10개)
2. "가격대는 어떤 포지션인가요?" — 럭셔리/프리미엄/미들/매스 (4개)
3. "몇 번째 수정인가요?" — 1차/2차/3차 이상 (3개)

**UX 주의사항**:
- AE가 사전에 입력해두고 클라이언트에게 링크를 보내는 시나리오도 지원해야 함 (URL 파라미터로 맥락 정보 프리셋 가능하게)
- "잘 모르겠어요" 선택지도 제공 (기본값 적용)
- 이 화면은 빠르게 넘겨야 함. 진짜 시작은 다음 질문 플로우.

### 3.4 화면 3: 질문 플로우 (핵심)

**목적**: 산파법으로 클라이언트의 의도를 구체화하는 단계.

**기본 원칙** (Phase 1, 2 공통):
- 한 번에 하나의 질문만 보여준다 (Progressive Disclosure)
- 모든 질문은 "지금 시안 대비" 프레임
- 전문 용어 절대 사용 금지. 클라이언트가 디자인을 몰라도 답할 수 있어야 함
- 총 질문 수는 **3~5개 이내**. 5개를 넘기면 안 됨
- 매 질문마다 진행률 표시 (예: 3/5)

**질문 유형**:

1. **텍스트 선택형**: 대비 질문 2개 선택지 + "둘 다 아닌데요" 옵션
   ```
   "묵직하고 전통 있는 느낌을 원하시나요,
    비워내서 여유로운 느낌을 원하시나요?"
   
   [묵직한 전통] [여유로운 비움] [둘 다 아닌데요]
   ```

2. **이미지 A/B 비교형**: 두 분기의 레퍼런스 이미지를 나란히 보여주고 선택
   ```
   "어느 쪽이 더 원하시는 방향에 가까우세요?"
   
   [이미지 A]  [이미지 B]  [둘 다 아닌데요]
   ```

3. **부정 확인형** (부정형 트리거 진입 시): 현재 시안의 문제 요소를 특정
   ```
   "딱딱하다고 느끼신 부분이 어디인가요?"
   
   [색감이 차가워요] [글씨가 딱딱해요] [전체 분위기가요] [잘 모르겠어요]
   ```

**"둘 다 아닌데요" 처리** (탈출 경로):
- 현재 대비 축과 다른 축의 질문으로 전환
- 2번 연속 "둘 다 아닌데요" 선택 시 → 이미지 A/B 비교로 전환
- 3번 연속 시 → "직접 설명해주셔도 좋아요" 자유 텍스트 입력 제공

**질문 엔진 — Phase 1 (정적 트리)**:

```
상태:
  - remaining_candidates: [분기 ID 배열]
  - eliminated: [분기 ID 배열]
  - answer_history: [{ question, answer, eliminated_branches }]
  - confidence: number (0~1)

초기 상태:
  - 트리거 매칭 결과의 candidate_branches로 시작
  - 맥락 변수에 따라 후보 순위 정렬 (boost/suppress)

매 질문 후:
  1. 선택된 방향과 반대되는 분기들을 remaining_candidates에서 제거
  2. anti_signals에 해당하는 분기 제거
  3. 남은 후보 중 가장 구분이 필요한 쌍의 contrast_question을 다음 질문으로 선택

수렴 조건 (질문 종료):
  - remaining_candidates가 2개 이하로 줄었을 때
  - 또는 연속 2회 같은 방향 선택 시
  - 또는 총 질문 5개 도달 시 (강제 수렴)

수렴 시 출력:
  - primary_branch: 최종 1순위 분기
  - secondary_branch: 2순위 참조 분기 (있을 경우)
  - eliminated_branches: 명시적으로 배제된 분기들
  - answer_log: 전체 질문-응답 이력
```

**질문 엔진 — Phase 2 (LLM 오케스트레이션)**:

Phase 1의 정적 로직을 LLM이 대체한다. 세부 설계는 섹션 7 참조.

### 3.5 화면 4: 클라이언트 확인

**목적**: "이렇게 이해했는데 맞으세요?" 확인 단계.

**구성 요소**:
- 해석 요약 (클라이언트용 언어): "말씀하신 '고급스러움'은 전통적 무게감보다는 비워내는 여유에 가까운 것 같아요."
- "이건 아닌" 방향도 함께 표시: "전통적이고 묵직한 고급스러움은 아닌 것 같아요."
- 레퍼런스 이미지 2~3장 (확정 분기의 대표 이미지)
- 안티 레퍼런스 1~2장 ("이런 느낌은 아닌 거죠?")
- [맞아요, 이 방향이에요] / [아니에요, 다시 해볼게요] 버튼

**해석 요약 생성**:
- Phase 1: 템플릿 기반. `description_client` 필드 조합.
- Phase 2: LLM이 응답 이력 기반으로 자연어 생성. 더 자연스럽고 맥락에 맞는 요약.

**"아니에요" 선택 시**:
- 질문 플로우로 돌아가되, 이전 답변 이력은 유지
- 추가 질문 1~2개로 보정

### 3.6 화면 5: 수정 방향 브리프 출력

**목적**: 디자이너 팀이 같은 방향을 보고 작업할 수 있는 최종 산출물.

**구성 요소** (시트 5 템플릿 기반):

**A. 헤더**
- 원문 피드백 (클라이언트가 입력한 그대로)
- 맥락 정보 (산업군 / 가격대 / 수정 차수)
- 생성 일시

**B. 해석 요약**
- 확정 분기 라벨 + 설명 (디자이너용)
- 배제 분기 라벨 + 이유
- 해석 근거 (질문-응답 로그 핵심 요약)

**C. 핵심 조정 포인트**
- 컬러 조정 방향
- 서체 조정 방향
- 레이아웃 조정 방향
- 이미지/사진 조정 방향
- 질감/텍스처 조정 방향

**D. 절대 하지 말 것**
- 배제 토큰 리스트
- 흔한 실수 경고 (혼동 분기와의 차이 강조)

**E. 레퍼런스**
- 참조 이미지 (분기 레퍼런스 풀에서)
- 안티 레퍼런스 이미지 ("이건 아닙니다")

**출력 형태**:
- 웹 페이지로 바로 열람 (고유 URL)
- PDF 다운로드
- 링크 복사 (디자이너 팀 공유용)

## 4. 기술 요구사항 — Phase 1 (정적 트리 MVP)

### 4.1 기술 스택

- **프론트엔드**: React (Next.js 권장) + TypeScript
- **스타일링**: Tailwind CSS
- **상태관리**: React Context 또는 Zustand (질문 엔진 상태)
- **데이터**: 온톨로지 JSON 파일 (static import, DB 불필요)
- **PDF 생성**: html2pdf 또는 react-pdf (브리프 다운로드용)
- **배포**: Vercel

### 4.2 데이터 흐름

```
ontology_v2.xlsx
  → 빌드 타임에 JSON 변환 (수동 또는 스크립트)
    → triggers.json      (시트 1)
    → branches.json      (시트 2)
    → tokens.json        (시트 3)
    → context.json       (시트 4)
    → brief_template.json (시트 5)

런타임:
  사용자 입력 → 트리거 매칭 → 질문 엔진 (상태 머신) → 브리프 생성 → 출력
```

### 4.3 핵심 모듈

**1. TriggerMatcher**
```
입력: 클라이언트 텍스트
출력: { matched_trigger, type, candidate_branches }
방식: 키워드 매칭 + fallback (빈출 표현 선택지 제시)
참고: Phase 2에서 LLM Orchestrator로 흡수됨. 삭제하지 않고 fallback으로 유지.
```

**2. QuestionEngine (상태 머신)**
```
상태: { remaining_candidates, eliminated, answer_history, confidence, question_count }
메서드:
  - getNextQuestion(): 남은 후보 중 가장 구분 필요한 쌍의 contrast_question 반환
  - submitAnswer(answer): 후보 분기 갱신, 배제 분기 업데이트
  - isConverged(): 수렴 여부 판정
  - getResult(): 최종 확정 분기 + 배제 분기 + 응답 로그
참고: Phase 2에서 LLM Orchestrator로 교체되지만, fallback으로 유지.
```

**3. BriefGenerator**
```
입력: { primary_branch, secondary_branch, eliminated_branches, answer_log, context }
출력: { client_summary, designer_brief, adjustment_tokens, never_do, references }
방식: Phase 1에서는 시트 2~5 데이터 조합 + 템플릿 렌더링. Phase 2에서는 LLM 자연어 생성 추가.
```

### 4.4 URL 구조

```
/ — 진입 화면
/session/:id — 질문 플로우 (세션별 고유 ID)
/session/:id/confirm — 클라이언트 확인 화면
/brief/:id — 브리프 열람 (디자이너 공유용 고유 URL)
/brief/:id/pdf — PDF 다운로드
```

### 4.5 맥락 정보 프리셋 (AE용)

AE가 클라이언트에게 링크 보낼 때 맥락 정보를 미리 채워서 보낼 수 있어야 함.

```
/session/new?industry=beauty&price=premium&stage=1st
```

## 5. 레퍼런스 이미지 관련

레퍼런스 이미지 풀 구축은 **별도 병렬 작업**으로 진행 중이다. 구현 시점에는 이미지가 완비되지 않을 수 있다.

### 5.1 대응 전략

- 이미지가 없는 분기는 **텍스트 선택형 질문만 사용** (이미지 A/B 비교 건너뜀)
- 이미지가 있는 분기부터 이미지 비교 활성화
- 브리프 출력의 레퍼런스 섹션도 이미지 유무에 따라 동적 처리

### 5.2 이미지 데이터 구조

```json
{
  "branch_id": "LUX-B",
  "reference_images": [
    { "url": "/images/ref/LUX-B-01.jpg", "caption": "Celine 2024 캠페인" },
    { "url": "/images/ref/LUX-B-02.jpg", "caption": "Aesop 매장 인테리어" },
    { "url": "/images/ref/LUX-B-03.jpg", "caption": "Bottega Veneta 패키지" }
  ],
  "anti_reference_images": [
    { "url": "/images/anti/LUX-B-anti-01.jpg", "caption": "대시보드 UI (CLN-A 혼동 주의)" },
    { "url": "/images/anti/LUX-B-anti-02.jpg", "caption": "장식 과잉 (LUX-D 혼동 주의)" }
  ]
}
```

### 5.3 이미지 없을 때 fallback

```
이미지 있을 때: [이미지 A] vs [이미지 B]
이미지 없을 때: "비워내서 여유로운 느낌" vs "정확하게 맞춘 깔끔한 느낌"
                 (description_client 필드 활용)
```

## 6. 디자인 가이드라인

### 6.1 톤 & 무드

이 앱의 톤은 **"잘 듣는 통역사"**. 권위적이지 않고, 클라이언트의 말을 존중하면서 부드럽게 명확화해주는 느낌.

- 질문은 항상 존댓말, 부드러운 어조
- "틀린 답은 없어요", "같이 찾아볼게요" 같은 안심 문구
- 디자인 용어는 클라이언트 화면에서 절대 노출하지 않음
- 디자이너 브리프에서만 전문 용어 사용

### 6.2 UI 원칙

- **한 화면에 하나의 행동만**. 질문 하나, 선택지 2~3개, 다음 버튼.
- **진행 감각**: 상단에 프로그레스 바 또는 스텝 인디케이터
- **되돌아가기 가능**: 이전 질문으로 돌아가서 답변 수정 가능
- **모바일 퍼스트**: 클라이언트는 모바일에서 볼 가능성 높음
- **이미지 A/B 비교**: 두 이미지를 나란히 (데스크톱) 또는 상하로 (모바일) 배치. 탭하면 선택.

### 6.3 시각적 방향

- 이 앱 자체가 특정 디자인 스타일을 강하게 가지면 안 됨 (레퍼런스 이미지 판단에 영향)
- 뉴트럴하고 깨끗한 UI. 콘텐츠(질문, 이미지)가 주인공.
- 배경 화이트/라이트그레이, 텍스트 다크, 강조색 1개 (브랜드 컬러 또는 뉴트럴 블루)

## 7. LLM 오케스트레이션 — Phase 2

> **이 섹션은 Phase 1(정적 트리 MVP) 구현 완료 후 실행한다.**
> Phase 1의 TriggerMatcher + QuestionEngine을 하나의 LLM Orchestrator로 교체한다.
> 프론트엔드 화면 구조는 변경 없음.

### 7.1 왜 필요한가

Phase 1의 정적 트리는 다음을 처리하지 못한다.

1. **트리거 매칭 실패**: "뭔가 좀 후지게 느껴져요", "경쟁사 A처럼은 안 됐으면" 같은 트리거 사전에 없는 표현
2. **교차 감성**: "고급스러운데 너무 올드하면 안 되고 약간 발랄했으면" — 단일 분기 선택으로 수렴 불가
3. **맥락 누적 해석**: 3번째 질문에서의 답변이 1번째 답변의 의미를 소급 변경하는 경우
4. **자연어 미세 뉘앙스**: "좀 더"와 "훨씬 더", "약간"과 "확 다르게"의 강도 차이

### 7.2 아키텍처 변경

```
Phase 1:
  클라이언트 입력 → TriggerMatcher (키워드) → QuestionEngine (정적 트리) → BriefGenerator (템플릿) → 출력

Phase 2:
  클라이언트 입력 → LLM Orchestrator (자연어 이해 + 동적 질문 + 분기 판정) → BriefGenerator (LLM 생성 + 토큰 조합) → 출력
  (실패 시)       → TriggerMatcher + QuestionEngine (fallback) →
```

### 7.3 백엔드 추가

Phase 1은 프론트엔드 only였지만, LLM 호출을 위해 API 레이어가 필요하다.

```
POST /api/session/start
  body: { feedback_text, context: { industry, price, stage } }
  response: { session_id, initial_analysis, first_question }

POST /api/session/:id/answer
  body: { answer }
  response: { updated_state, next_question } 또는 { converged: true, result }

POST /api/brief/:id/generate
  body: { session_result }
  response: { client_summary, designer_brief }
```

### 7.4 LLM 시스템 프롬프트 구조

LLM에게 매 호출마다 다음을 컨텍스트로 제공한다.

```
[시스템 프롬프트]

당신은 브랜드 디자인 에이전시의 피드백 번역 시스템입니다.
클라이언트의 모호한 수정 피드백을 구체화하는 질문을 생성합니다.

## 역할
- 클라이언트에게 디자인 전문 용어를 절대 사용하지 않는다
- 질문은 항상 "지금 시안 대비" 프레임으로 한다
- 한 번에 하나의 질문만 한다
- 총 질문 수는 5개를 넘기지 않는다

## 온톨로지 데이터
[여기에 branches.json 전체를 삽입 — 37개 분기의 전체 데이터]

## 맥락 변수
[여기에 context.json + 현재 세션의 산업군/가격대/프로젝트단계 삽입]

## 현재 세션 상태
- 원문 피드백: "{클라이언트 입력 원문}"
- 질문-응답 이력: [{이전 질문과 답변들}]
- 남은 후보 분기: [분기 ID 배열]
- 배제된 분기: [분기 ID 배열 + 배제 이유]
- 현재 질문 횟수: N/5
```

### 7.5 LLM 호출 흐름

**호출 1: 초기 분석**

```
입력: 클라이언트 원문 피드백 + 맥락 변수
요청: 다음을 JSON으로 응답하라.
  1. 피드백 유형 판정 (directional / negative / ambiguous)
  2. 관련 감성 축 식별
  3. 초기 후보 분기 목록 (온톨로지에서)
  4. 초기 배제 분기 목록 (안티 시그널 기반)
  5. 첫 번째 질문 (클라이언트용 비전문가 언어)
  6. 질문의 선택지 2~3개
```

응답 예시:
```json
{
  "feedback_type": "directional",
  "axis": "품격/가치",
  "candidates": ["LUX-A", "LUX-B", "LUX-C", "LUX-D"],
  "eliminated": [],
  "question": "고급스러움에도 여러 느낌이 있어요. 지금 시안에서 어떤 방향을 더 살리고 싶으세요?",
  "options": [
    {"label": "묵직하고 전통 있는 느낌", "direction": "LUX-A"},
    {"label": "비워내서 여유로운 느낌", "direction": "LUX-B,LUX-C"},
    {"label": "화려하고 풍성한 느낌", "direction": "LUX-D"}
  ]
}
```

**호출 2~N: 후속 질문 생성**

```
입력: 이전까지의 전체 세션 상태 + 방금 클라이언트의 답변
요청: 다음을 JSON으로 응답하라.
  1. 이번 답변으로 배제할 분기 + 이유
  2. 남은 후보 분기 업데이트
  3. 수렴 여부 판정 (converged: true/false)
  4. 수렴이 아니면: 다음 질문 + 선택지
  5. 수렴이면: 최종 판정 (primary_branch, secondary_branch, reasoning)
```

**호출 최종: 브리프 생성**

```
입력: 확정 분기 + 배제 분기 + 전체 응답 이력 + 온톨로지 토큰 데이터
요청: 다음을 생성하라.
  1. 클라이언트 확인용 해석 요약 (비전문가 언어, 2~3문장)
  2. 디자이너용 해석 요약 (전문 용어, 근거 포함)
  3. 조정 포인트 요약 (온톨로지 토큰을 현재 맥락에 맞게 조정)
  4. 흔한 실수 경고 (혼동 분기 기반)
```

### 7.6 LLM 응답 포맷 강제

모든 LLM 호출에서 응답은 **JSON only**로 강제한다. 프리앰블, 마크다운 백틱 없이 순수 JSON만 반환하도록 시스템 프롬프트에 명시. 파싱 실패 시 재시도 로직 포함.

### 7.7 엔트로피 최소화 전략

시스템 프롬프트에 포함할 지시:
```
다음 질문을 생성할 때, 남은 후보 분기들 사이에서 가장 큰 차이를 드러내는 축을 선택하라.
이미 확정된 축은 다시 묻지 마라.
예: 온도감 축이 이미 "따뜻한" 쪽으로 확정되었으면, 다음은 아직 미확정인 "권위↔친근" 축이나 "장식↔절제" 축을 물어라.
```

### 7.8 교차 감성 처리

정적 트리에서 불가능했던 핵심 기능.

```
시스템 프롬프트 지시:
클라이언트의 피드백이 두 개 이상의 감성 축에 걸칠 경우,
단일 분기 선택이 아니라 가중합으로 판정하라.

예: "고급스러운데 약간 발랄한"
→ primary: LUX-B (미니멀 럭셔리) 0.6
→ secondary: VIV-A (팝 에너지) 0.2 — 단, 에너지 요소만 차용
→ 배제: LUX-A (올드머니) — "발랄"과 충돌

브리프 생성 시 가중합 로직:
- primary 분기의 토큰을 기본으로 하되
- secondary 분기에서 차용할 요소를 명시적으로 지정
- 충돌하는 토큰은 primary 우선
```

### 7.9 시스템 프롬프트 전문 (초안)

```
당신은 DCTS(Design Communication Translation System)의 질문 엔진입니다.

## 당신의 역할
브랜드 디자인 에이전시의 클라이언트가 수정 피드백을 보냈습니다.
클라이언트의 머릿속에는 분명히 원하는 그림이 있지만, 디자인 어휘의 한계로 모호하게 전달됩니다.
당신은 산파법 질문을 통해 그 그림을 구체화하는 역할입니다.

## 핵심 원칙
1. 디자인 전문 용어를 절대 사용하지 않는다. 클라이언트는 디자인을 모른다.
2. 모든 질문은 "지금 시안 대비" 프레임이다. 절대 좌표가 아니라 상대적 방향을 묻는다.
3. 한 번에 하나의 질문만. 선택지는 2~3개. "둘 다 아닌데요" 옵션 항상 포함.
4. 총 질문은 5개 이내. 매 질문은 남은 후보 분기를 가장 효율적으로 줄이는 것을 선택.
5. 이전 답변이 이후 질문의 방향을 결정한다. 이미 확정된 축은 다시 묻지 않는다.
6. 질문의 톤은 부드럽고 안심시키는 느낌. "틀린 답은 없어요" 뉘앙스.

## 온톨로지 데이터
아래는 이 시스템의 감성 해석 분기 데이터입니다. 37개 분기, 12개 감성 축.
이 데이터 안에서만 판단합니다. 데이터에 없는 분기를 만들어내지 않습니다.

{branches_json}

## 맥락 변수 가중치
{context_json}

## 응답 형식
반드시 JSON만 응답하라. 마크다운 백틱, 설명문, 프리앰블 금지.
```

### 7.10 가드레일

1. **온톨로지 외부의 분기를 만들어내지 않는다**: 37개 분기 ID 이외의 것을 primary/secondary로 지정하면 안 됨. 응답 검증 로직에서 체크.
2. **디자인 전문 용어를 질문에 사용하지 않는다**: "세리프", "그리드", "채도" 같은 단어가 질문이나 선택지에 포함되면 리젝트하고 재생성.
3. **5개 질문 이내 수렴**: question_count가 5에 도달하면 강제 수렴.
4. **JSON 포맷 강제**: 파싱 실패 시 최대 2회 재시도. 3회 실패 시 Phase 1 정적 트리 fallback.

### 7.11 정적 트리 fallback

LLM 호출 실패, 타임아웃(5초), 파싱 에러 시 Phase 1의 정적 트리 로직으로 자동 전환. 클라이언트는 차이를 느끼지 못함. **Phase 1의 TriggerMatcher + QuestionEngine 코드를 삭제하지 않고 반드시 유지**.

### 7.12 프론트엔드 변경사항

화면 구조는 동일. 바뀌는 건:
- QuestionEngine의 데이터 소스: 로컬 JSON → API 호출
- 질문 선택지: 정적 2~3개 → LLM 동적 생성 2~3개
- 브리프 해석 요약: 템플릿 조합 → LLM 생성 자연어
- 추가: 질문 전환 시 로딩 인디케이터 ("생각하고 있어요...")

### 7.13 LLM 선택 & 비용

- **권장**: Claude Sonnet — 비용/성능 밸런스. 온톨로지 전체를 컨텍스트에 넣어도 충분한 윈도우.
- **대안**: GPT-4o
- 온톨로지 JSON 전체 크기: 약 15,000~20,000 토큰
- 세션당 호출 횟수: 약 4~7회 (초기 분석 1 + 질문 2~5 + 브리프 생성 1)
- 세션당 예상 비용: Sonnet 기준 약 $0.10~0.15
- 월 100세션 가정: 약 $10~15

### 7.14 프롬프트 최적화

- 후속 호출에서는 남은 후보 분기의 데이터만 넣어 토큰 절약 가능 (초기 분석에서는 전체 필요)
- 5개 질문이므로 응답 이력 전체를 넣어도 토큰 부담 적음
- 맥락 변수는 부스트/서프레스 분기 ID만 넣으면 충분

### 7.15 Phase 2 구현 순서

Phase 1 완료 후 다음 순서로 진행:

1. **API 레이어 구축** (2~3일): Next.js API Routes, LLM API 연동, 세션 상태 관리
2. **프롬프트 작성 & 테스트** (3~5일): 10+ 시나리오 수동 테스트, JSON 파싱 안정성, 가드레일 검증
3. **프론트엔드 연동** (2~3일): API 교체, 로딩 인디케이터, fallback 전환 로직
4. **A/B 테스트** (1주): 정적/LLM 50:50 라우팅, 품질 및 비용 비교

예상 총 소요: 약 2~3주

### 7.16 Phase 2 검증 지표

정적 트리와 LLM의 A/B 비교:
- 동일 입력 → 동일 또는 더 나은 분기 판정이 나오는지
- 평균 질문 수 감소 여부
- "둘 다 아닌데요" 빈도 감소 여부 (질문 품질 향상)
- 교차 감성 케이스에서 합리적인 가중합이 나오는지
- 클라이언트 완주율, "맞아요" 비율, 디자이너 방향 일치율

## 10. 주의사항 & 핵심 리스크

### 10.1 가장 중요한 원칙

**이 시스템의 주 목적은 클라이언트의 머릿속에 있는 그림을 산파해내는 것이다.** 디자이너 편의는 부수 효과. 모든 설계 판단에서 "클라이언트가 이 질문에 쉽게 답할 수 있는가?"가 최우선 기준.

### 10.2 핵심 리스크

1. **클라이언트 이탈**: 질문이 5개를 넘거나, 질문이 어렵거나, 가치를 못 느끼면 이탈함. 매 질문이 "아, 그래 내가 원한 게 이거야!"라는 발견의 순간을 줘야 함.

2. **트리거 매칭 실패** (Phase 1): 클라이언트가 트리거 사전에 없는 표현을 쓸 때. 빈출 표현 선택지 fallback으로 대응하되, 매칭 실패 로그를 수집해서 Phase 2의 LLM 전환 판단 근거로 활용.

3. **브리프를 디자이너가 안 봄**: 기존 워크플로우에 끼어들지 못하면 쓰이지 않음. 링크 공유 + PDF로 최소한 전달 가능하게.

4. **온톨로지 품질**: 시스템의 가치는 코드가 아니라 데이터 품질에 달려있음. 잘못된 대비 질문, 부정확한 토큰 매핑은 코드로 커버 불가.

5. **LLM 가드레일 실패** (Phase 2): 온톨로지 외부의 분기 생성, 전문 용어 노출, JSON 파싱 실패. 정적 트리 fallback으로 안전망 확보.
