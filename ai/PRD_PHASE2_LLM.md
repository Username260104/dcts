# PRD 확장: Phase 2 — LLM 오케스트레이션 통합

## 문서 정보

- **선행 조건**: PRD.md (MVP v1.0) 구현 완료
- **목적**: 정적 의사결정 트리를 LLM 기반 동적 질문 엔진으로 교체
- **핵심 변경**: TriggerMatcher와 QuestionEngine을 LLM으로 교체. BriefGenerator에 자연어 해석 요약 생성 추가.

---

## 1. 왜 이 확장이 필요한가

MVP의 정적 트리는 다음을 처리하지 못한다.

1. **트리거 매칭 실패**: 클라이언트가 "뭔가 좀 후지게 느껴져요", "경쟁사 A처럼은 안 됐으면" 같은 트리거 사전에 없는 표현을 쓸 때
2. **교차 감성**: "고급스러운데 너무 올드하면 안 되고 약간 발랄했으면" — 단일 분기 선택으로 수렴 불가
3. **맥락 누적 해석**: 3번째 질문에서의 답변이 1번째 답변의 의미를 소급 변경하는 경우
4. **자연어 미세 뉘앙스**: "좀 더"와 "훨씬 더", "약간"과 "확 다르게"의 강도 차이

LLM은 온톨로지 데이터를 컨텍스트로 갖고 있으면서, 매 응답마다 남은 후보 분기의 확정/미확정 영역을 판단하고, 가장 효율적인 다음 질문을 동적으로 생성한다.

---

## 2. 아키텍처 변경

### 2.1 MVP 구조 (현재)

```
클라이언트 입력
  → TriggerMatcher (키워드 매칭)
  → QuestionEngine (정적 분기 트리)
  → BriefGenerator (템플릿 조합)
  → 브리프 출력
```

### 2.2 Phase 2 구조 (변경 후)

```
클라이언트 입력
  → LLM Orchestrator (자연어 이해 + 동적 질문 생성 + 분기 판정)
  → BriefGenerator (LLM 해석 요약 생성 + 토큰 조합)
  → 브리프 출력
```

TriggerMatcher와 QuestionEngine이 하나의 LLM Orchestrator로 통합된다. 프론트엔드 화면 구조(진입→맥락→질문→확인→브리프)는 동일하게 유지. 바뀌는 건 질문 생성과 분기 판정의 백엔드 로직.

## 3. LLM Orchestrator 상세 설계

### 3.1 시스템 프롬프트 구조

LLM에게 매 호출마다 다음을 컨텍스트로 제공한다.

```
[시스템 프롬프트]

당신은 브랜드 디자인 에이전시의 피드백 번역 시스템입니다.
클라이언트의 모호한 수정 피드백을 구체화하는 질문을 생성합니다.

## 역할
- 클라이언트에게 디자인 전문 용어를 절대 사용하지 않는다
- 질문은 항상 "지금 시안 대비" 프레임으로 한다
- 한 번에 하나의 질문만 한다
- 총 질문 수는 5개를 넘기지 않는다

## 온톨로지 데이터
[여기에 branches.json 전체를 삽입 — 37개 분기의 전체 데이터]

## 맥락 변수
[여기에 context.json + 현재 세션의 산업군/가격대/프로젝트단계 삽입]

## 현재 세션 상태
- 원문 피드백: "{클라이언트 입력 원문}"
- 질문-응답 이력: [{이전 질문과 답변들}]
- 남은 후보 분기: [분기 ID 배열]
- 배제된 분기: [분기 ID 배열 + 배제 이유]
- 현재 질문 횟수: N/5
```

### 3.2 LLM 호출 흐름

**호출 1: 초기 분석**

```
입력: 클라이언트 원문 피드백 + 맥락 변수
요청: 다음을 JSON으로 응답하라.
  1. 피드백 유형 판정 (directional / negative / ambiguous)
  2. 관련 감성 축 식별
  3. 초기 후보 분기 목록 (온톨로지에서)
  4. 초기 배제 분기 목록 (안티 시그널 기반)
  5. 첫 번째 질문 (클라이언트용 비전문가 언어)
  6. 질문의 선택지 2~3개
```

응답 예시:
```json
{
  "feedback_type": "directional",
  "axis": "품격/가치",
  "candidates": ["LUX-A", "LUX-B", "LUX-C", "LUX-D"],
  "eliminated": [],
  "question": "고급스러움에도 여러 느낌이 있어요. 지금 시안에서 어떤 방향을 더 살리고 싶으세요?",
  "options": [
    {"label": "묵직하고 전통 있는 느낌", "direction": "LUX-A"},
    {"label": "비워내서 여유로운 느낌", "direction": "LUX-B,LUX-C"},
    {"label": "화려하고 풍성한 느낌", "direction": "LUX-D"}
  ]
}
```

**호출 2~N: 후속 질문 생성**

```
입력: 이전까지의 전체 세션 상태 + 방금 클라이언트의 답변
요청: 다음을 JSON으로 응답하라.
  1. 이번 답변으로 배제할 분기 + 이유
  2. 남은 후보 분기 업데이트
  3. 수렴 여부 판정 (converged: true/false)
  4. 수렴이 아니면: 다음 질문 + 선택지
  5. 수렴이면: 최종 판정 (primary_branch, secondary_branch, reasoning)
```

**호출 최종: 브리프 생성**

```
입력: 확정 분기 + 배제 분기 + 전체 응답 이력 + 온톨로지 토큰 데이터
요청: 다음을 생성하라.
  1. 클라이언트 확인용 해석 요약 (비전문가 언어, 2~3문장)
  2. 디자이너용 해석 요약 (전문 용어, 근거 포함)
  3. 조정 포인트 요약 (온톨로지 토큰을 현재 맥락에 맞게 조정)
  4. 흔한 실수 경고 (혼동 분기 기반)
```

### 3.3 LLM 응답 포맷 강제

모든 LLM 호출에서 응답은 **JSON only**로 강제한다. 프리앰블, 마크다운 백틱 없이 순수 JSON만 반환하도록 시스템 프롬프트에 명시. 파싱 실패 시 재시도 로직 포함.

### 3.4 엔트로피 최소화 전략

LLM에게 "다음 질문은 남은 후보 분기를 가장 효율적으로 줄이는 질문을 선택하라"고 지시한다.

시스템 프롬프트에 포함할 지시:
```
다음 질문을 생성할 때, 남은 후보 분기들 사이에서 가장 큰 차이를 드러내는 축을 선택하라.
이미 확정된 축은 다시 묻지 마라.
예: 온도감 축이 이미 "따뜻한" 쪽으로 확정되었으면, 다음은 아직 미확정인 "권위↔친근" 축이나 "장식↔절제" 축을 물어라.
```

### 3.5 교차 감성 처리

정적 트리에서 불가능했던 핵심 기능. LLM은 "고급스러운데 약간 발랄한"을 이렇게 처리한다.

```
시스템 프롬프트 지시:
클라이언트의 피드백이 두 개 이상의 감성 축에 걸칠 경우,
단일 분기 선택이 아니라 가중합으로 판정하라.

예: "고급스러운데 약간 발랄한"
→ primary: LUX-B (미니멀 럭셔리) 0.6
→ secondary: VIV-A (팝 에너지) 0.2 — 단, 에너지 요소만 차용
→ 배제: LUX-A (올드머니) — "발랄"과 충돌

브리프 생성 시 가중합 로직:
- primary 분기의 토큰을 기본으로 하되
- secondary 분기에서 차용할 요소를 명시적으로 지정
- 충돌하는 토큰은 primary 우선
```

## 4. 구현 가이드

### 4.1 백엔드 추가

MVP는 프론트엔드 only (정적 JSON)였지만, LLM 호출을 위해 API 레이어가 필요하다.

```
POST /api/session/start
  body: { feedback_text, context: { industry, price, stage } }
  response: { session_id, initial_analysis, first_question }

POST /api/session/:id/answer
  body: { answer }
  response: { updated_state, next_question } 또는 { converged: true, result }

POST /api/brief/:id/generate
  body: { session_result }
  response: { client_summary, designer_brief }
```

### 4.2 LLM 선택

- **권장**: Claude Sonnet — 비용/성능 밸런스. 온톨로지 전체를 컨텍스트에 넣어도 충분한 컨텍스트 윈도우.
- **대안**: GPT-4o — 동등한 성능.
- 온톨로지 JSON 전체 크기 예상: 약 15,000~20,000 토큰. 매 호출마다 포함해도 비용 부담 적음.

### 4.3 응답 지연 관리

- LLM 응답 시간: 1~3초 예상
- 클라이언트 경험: 질문 전환 시 "생각하고 있어요..." 로딩 인디케이터 표시
- 스트리밍 불필요 (JSON 응답이므로 완성 후 한 번에 렌더)
- 타임아웃 5초. 초과 시 해당 질문을 정적 트리 fallback으로 대체.

### 4.4 정적 트리 fallback

LLM 호출 실패, 타임아웃, 파싱 에러 시 MVP의 정적 트리 로직으로 자동 전환. 클라이언트는 차이를 느끼지 못함. 이를 위해 MVP의 QuestionEngine 코드를 삭제하지 않고 유지.

### 4.5 프론트엔드 변경 최소화

화면 구조는 MVP와 동일. 바뀌는 건:
- QuestionEngine의 데이터 소스가 로컬 JSON → API 호출로 전환
- 질문 선택지가 정적 2~3개 → LLM이 동적으로 생성한 2~3개로 전환
- 브리프의 해석 요약이 템플릿 조합 → LLM 생성 자연어로 전환
- 로딩 인디케이터 추가

## 5. 프롬프트 엔지니어링 상세

### 5.1 시스템 프롬프트 전문 (초안)

```
당신은 DCTS(Design Communication Translation System)의 질문 엔진입니다.

## 당신의 역할
브랜드 디자인 에이전시의 클라이언트가 수정 피드백을 보냈습니다.
클라이언트의 머릿속에는 분명히 원하는 그림이 있지만, 디자인 어휘의 한계로 모호하게 전달됩니다.
당신은 산파법 질문을 통해 그 그림을 구체화하는 역할입니다.

## 핵심 원칙
1. 디자인 전문 용어를 절대 사용하지 않는다. 클라이언트는 디자인을 모른다.
2. 모든 질문은 "지금 시안 대비" 프레임이다. 절대 좌표가 아니라 상대적 방향을 묻는다.
3. 한 번에 하나의 질문만. 선택지는 2~3개. "둘 다 아닌데요" 옵션 항상 포함.
4. 총 질문은 5개 이내. 매 질문은 남은 후보 분기를 가장 효율적으로 줄이는 것을 선택.
5. 이전 답변이 이후 질문의 방향을 결정한다. 이미 확정된 축은 다시 묻지 않는다.
6. 질문의 톤은 부드럽고 안심시키는 느낌. "틀린 답은 없어요" 뉘앙스.

## 온톨로지 데이터
아래는 이 시스템의 감성 해석 분기 데이터입니다. 37개 분기, 12개 감성 축.
이 데이터 안에서만 판단합니다. 데이터에 없는 분기를 만들어내지 않습니다.

{branches_json}

## 맥락 변수 가중치
{context_json}

## 응답 형식
반드시 JSON만 응답하라. 마크다운 백틱, 설명문, 프리앰블 금지.
```

### 5.2 가드레일

LLM이 벗어나면 안 되는 경계:

1. **온톨로지 외부의 분기를 만들어내지 않는다**: 37개 분기 ID 이외의 것을 primary/secondary로 지정하면 안 됨. 응답 검증 로직에서 체크.
2. **디자인 전문 용어를 질문에 사용하지 않는다**: "세리프", "그리드", "채도" 같은 단어가 질문이나 선택지에 포함되면 리젝트하고 재생성.
3. **5개 질문 이내 수렴**: question_count가 5에 도달하면 강제 수렴. LLM에게 "지금까지의 정보로 최선의 판정을 내려라"고 지시.
4. **JSON 포맷 강제**: 파싱 실패 시 최대 2회 재시도. 3회 실패 시 정적 트리 fallback.

### 5.3 프롬프트 최적화 포인트

- 온톨로지 JSON을 매 호출 넣는 대신, 현재 남은 후보 분기의 데이터만 넣으면 토큰 절약 가능. 단, 초기 분석 호출에서는 전체 필요.
- 응답 이력이 길어지면 요약해서 넣는 것 고려. 5개 질문이므로 MVP에서는 전체 이력을 넣어도 토큰 부담 적음.
- 맥락 변수 가중치는 부스트/서프레스 분기 ID만 넣으면 충분. 전체 테이블 불필요.

## 6. 테스트 및 검증

### 6.1 LLM 품질 검증

정적 트리 MVP의 테스트 케이스를 그대로 LLM 버전에 돌려서 비교한다.

- 동일 입력 → 동일 또는 더 나은 분기 판정이 나오는지
- 질문 수가 줄어드는지 (정적 트리 대비)
- "둘 다 아닌데요" 빈도가 줄어드는지 (질문 품질 향상 확인)
- 교차 감성 케이스에서 합리적인 가중합이 나오는지

### 6.2 정적 트리와의 A/B 비교

일정 기간 동안 세션의 50%는 정적 트리, 50%는 LLM으로 라우팅해서 비교.

- 클라이언트 완주율
- "맞아요" 비율 (확인 화면)
- 디자이너 방향 일치율
- 평균 질문 수

### 6.3 비용 모니터링

- 세션당 LLM 호출 횟수: 약 4~7회 (초기 분석 1 + 질문 2~5 + 브리프 생성 1)
- 세션당 예상 토큰: 입력 ~25,000 + 출력 ~3,000 = ~28,000 토큰
- 세션당 예상 비용: Sonnet 기준 약 $0.10~0.15
- 월 100세션 가정: 약 $10~15

## 7. 구현 순서

이 확장은 MVP 완성 후 다음 순서로 진행한다.

**Step 1: API 레이어 구축** (2~3일)
- Next.js API Routes 또는 별도 서버
- Anthropic API (또는 OpenAI API) 연동
- 세션 상태 관리 (in-memory 또는 Redis)
- 정적 트리 fallback 로직 유지

**Step 2: 시스템 프롬프트 작성 및 테스트** (3~5일)
- 섹션 5의 프롬프트를 실제 API로 테스트
- 온톨로지 JSON을 컨텍스트에 넣고 10+ 시나리오 수동 테스트
- JSON 파싱 안정성 확인
- 가드레일 검증 (전문 용어 필터, 분기 ID 검증)

**Step 3: 프론트엔드 연동** (2~3일)
- QuestionEngine의 데이터 소스를 API로 교체
- 로딩 인디케이터 추가
- fallback 전환 로직 구현
- 브리프 생성부 LLM 응답으로 교체
